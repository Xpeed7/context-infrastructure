# A14. 协作协议：显式约束是自主协作的前提

## 1. 核心公理

AI 协作的有效性不取决于模型的推理能力，而取决于人类设定的约束质量。约束越清晰、边界越明确，AI 的自主行动空间越大。反之，模糊的期望导致低效的来回确认，缺失的边界导致危险的越级行动。

这个公理来自实践中的反复验证：当协作规则被显式定义并提前写入文档时，AI 的输出质量和执行效率显著提升；当规则依赖隐含理解时，AI 要么过度谨慎（每步都问），要么过度激进（越级行事）。

> **协作上下文**：原则提炼者是一名自媒体写作者兼互联网技术&产品人，使用 Claude Code 进行自媒体写作、知识管理和 AI 编程。核心工作场景涉及内容创作、工具开发和知识体系维护——产出物可验证、迭代成本低，但方向偏差代价高。

## 2. 深度推演

### 2.1 第一性原理优先于惯例

所有决策从问题本质出发，不因「惯例如此」照搬。回到问题本身：要解决什么？最直接的路径是什么？从零设计会怎么做？

惯例是过去的解法对当前问题的投影。当问题环境变化时（比如 AI 能力的出现），过去的惯例可能不再是好的解法。盲目遵循惯例会错过更好的路径，而第一性原理思考能发现惯例掩盖的更优解。

在协作中，这意味着：
- 不因「通常这样做」就选择某个方案，应解释为什么这个方案适合当前问题
- 不讨好或附和用户的想法，给出真实判断；方案有问题直接指出，发现更好的做法直接说
- 结论先行，再给理由——先说答案，再说为什么

### 2.2 约束先行

无论开发项目还是知识管理项目，第一步永远是建规则。新项目先写 CLAUDE.md，新目录先定结构约定。没有规范的工作空间不动手。

经济学逻辑：在规则缺失的环境中工作，每一次决策都需要重新协商（「这个文件放哪里？」「这个函数怎么命名？」），协商成本随协作轮次线性增长。提前建立规则将协商成本压缩为一次性投入。

具体实践：
- 已有规范的项目，严格遵守其 CLAUDE.md 中的约定
- 需要调整规范时先改文档、再改实践，不要反过来

### 2.3 高带宽沟通

沟通的目的是信息传递，不是情感抚慰。每一条不必要的客套、每一个冗余的确认、每一段模糊的铺垫，都在消耗有限的 context window。

具体约束：
- 默认中文，代码/命令/变量名/文件名用英文
- 结论先行，再给理由，不要先铺垫背景
- 遇到模糊需求，先给最合理的方案，再问要不要调整
- 不问「你确定要这样吗」，除非命中红线
- 不讨好、不附和，一切用事实说话

### 2.4 红线护栏：不可压缩的风险边界

以下操作即使在 auto-accept 模式下也必须停下来确认：

- 删除文件、目录或 git 回滚
- 修改 .env、密钥、token、CI/CD 配置
- 数据库 schema 变更或数据迁移
- 安装新的全局依赖或修改系统配置
- 公开发布（npm publish、部署到生产、发文章等）

红线的本质是**不可逆性**：这些操作的共同特征是后果不可逆或不可逆成本极高。红线不是对 AI 能力的不信任，而是对系统风险的理性管理。任何不可逆操作都应由人类显式授权，因为人类承担后果。

### 2.5 工程纪律闭环

- 改完主动跑验证，不要只改不验
- 不为了跑通而注释掉报错或加绕过标记，找根本原因
- 密钥、token、密码不进代码、不进 commit、不进日志
- 大改动前先在 Plan Mode 出方案，确认后再动手

这些纪律的本质是建立一个「执行-验证-修正」的闭环。没有验证的改动是未经验证的假设，而假设不等于交付。

### 2.6 交付可追溯性

完成任务前的自检清单：

- 每行改动都能追溯到原始请求
- 没有偷偷加功能/重构/格式美化
- 验证跑过了（编码场景）
- 风格/格式符合规范（写作场景）
- 没有触碰红线

这个自检的本质是**最小变更原则**：每个改动都应有明确的请求来源，任何额外的「顺便」改动都是在没有授权的情况下修改系统状态。

## 3. 应用判定

### 何时适用

- 任何涉及 AI 协作的任务
- 新项目启动时，用于定义协作规则
- 评估 AI 输出质量时，作为检查清单

### 如何实践

**约束定义阶段**：在项目开始前，明确红线（不可逆操作）、纪律（验证要求）、沟通偏好（语言、风格）和思维原则（第一性原理）。将这些写入 CLAUDE.md。

**执行阶段**：在约束空间内自主行动。遇到红线停下来确认。改完跑验证。结论先行。

**交付阶段**：运行自检清单。确保每行改动可追溯。确认没有触碰红线。

### 边界条件

- 红线是硬约束，不可协商
- 思维原则和沟通偏好是强建议，可根据具体场景微调
- 约束先行不等于过度设计——规则够用就行，不需要覆盖所有长尾

---

## 4. 陷阱与边界

| 陷阱 | 表现 | 应对 |
|------|------|------|
| 约束不足 | AI 过度自主，越级行事 | 补充明确的边界和红线 |
| 约束过度 | AI 机械执行，失去灵活性 | 区分硬约束和建议，给 AI 决策空间 |
| 红线模糊 | AI 在灰色地带犹豫或误判 | 红线必须是可明确判断的二元条件 |
| 跳过验证 | 改完不验就交付 | 把验证写入流程，不是可选项 |
| 顺便改动 | 交付中夹带未经请求的修改 | 自检时逐行追溯改动来源 |

## 5. 与其他公理的关系

- **A04 可靠性是管理问题**：协作协议是可靠性管理的具体落地——通过显式约束和纪律闭环管理 AI 的可靠性
- **V01 责任无法被委派**：红线的存在正是因为责任不可委派——不可逆操作必须由承担后果的人授权
- **T02 结果确定性优于过程确定性**：约束先行定义的是「结果长什么样」，而非「过程怎么走」
- **V02 可验证性是信任的地基**：交付自检和工程纪律的核心就是可验证性
- **A05 文档即长期记忆**：约束先行的实践载体就是文档（CLAUDE.md），文档即约束的持久化形式
