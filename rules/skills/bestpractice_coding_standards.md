# Coding 规范与纪律

## 元数据

- **类型**: BestPractice
- **适用场景**: 任何涉及编写、修改、调试代码的任务
- **创建日期**: 2026-05-27
- **来源**: 从全局 CLAUDE.md 迁移，源自长期 AI 编程实践

## 目标

用最少的代码、最精准的改动、可验证的标准完成任务——不做多余的事，不漏该做的事。

## 边界

- 适用于所有编码任务，包括新写代码、修改现有代码、调试
- 不覆盖项目管理、写作、知识管理等非编码场景（那些见全局 CLAUDE.md）

## 核心约束

### 编码前思考

不要假设。不要隐藏困惑。呈现权衡。

- 明确说明假设 — 不确定就问，不猜
- 呈现多种解释 — 有歧义时不默默选择
- 适时提出异议 — 有更简单的做法直接说
- 困惑时停下来 — 指出不清楚的地方要求澄清

### 简洁优先

用最少的代码解决问题。对抗过度工程：

- 不添加要求之外的功能
- 不为一次性代码创建抽象
- 不添加未要求的"灵活性"或"可配置性"
- 不为不可能发生的场景做错误处理
- 200 行能写成 50 行就重写

检验标准：资深工程师会觉得过于复杂吗？如果是，简化。

### 精准修改

只碰必须碰的。只清理自己造成的混乱。

- 不"改进"相邻的代码、注释或格式
- 不重构没坏的东西
- 匹配现有风格，即使你更倾向不同写法
- 注意到无关死代码可以提，但不要删
- 因你的改动产生的孤儿代码（无用的导入/变量/函数），删掉
- 不要删除预先存在的死代码，除非被要求

检验标准：每一行修改都能直接追溯到用户的请求。

### 目标驱动执行

定义成功标准，循环验证直到达成。

将指令式任务转化为可验证的目标：

| 不要这样做 | 转化为 |
| --- | --- |
| "添加验证" | "为无效输入编写测试，然后让它们通过" |
| "修复 bug" | "编写重现 bug 的测试，然后让它通过" |
| "重构 X" | "确保重构前后测试都能通过" |

多步骤任务说明简短计划：`[步骤] → 验证: [检查]`。强成功标准让 LLM 能独立循环执行；弱标准（"让它工作"）需要不断澄清。

## 调试纪律

- 涉及外部 API/协议/SDK 的问题，**第一步**找官方文档或可运行的参考实现（GitHub demo、SDK 源码），逐行对比 headers、body、参数生成逻辑，不靠猜测和加日志摸索
- 对比必须逐字段完整，包括 headers、body 结构、工具函数内部实现——"看起来差不多"不算对比过
- 所有差异一次性修完再测试，不要改一个点就让用户验证，尤其是高成本操作（扫码、重新部署、重启服务）
- 禁止无证据归因——"可能是 X 导致的"必须先验证再下结论，不把猜测当方向让用户配合排查
- 调试超过两轮未解决，立刻停下来换策略：找参考实现、搜索已知问题、对比工作版本和当前版本的 diff，不要继续同一个方向加日志

## 已知陷阱

来自真实失败记录，禁止重犯：

- **2026-04-22**：用户说「固定 index 放最新内容」，未经确认直接改代码，并连带部署到 CF 生产环境。**越界两条红线**：改代码前没说明方案等确认；CF 部署没单独询问。教训：即使需求看似明确，改代码前必须说明方案等确认；涉及生产环境的操作必须单独确认。
- **2026-04-28**：调试 iLink sendmessage 发送失败，靠猜测反复试错（怀疑 token 过期、baseurl 不对、watch 重启异常），让用户扫码 6 次以上，最终才去对比 demo 源码发现是 headers 和 UIN 生成方式的差异。教训：外部协议问题第一步找参考实现逐行对比，不要让用户配合无证据的猜测反复验证。
- **2026-05-27**：首次提交代码时未检查 git config，导致 committer 显示为系统默认的 hostname 格式而非用户指定名称。教训：提交前先确认 `git config user.name` 和 `git config user.email` 正确，新项目首次提交必检。

## 验收标准

每次编码任务完成后逐条检查：

- 每行改动都能追溯到用户的请求（精准修改）
- 没有添加要求之外的功能、抽象或错误处理（简洁优先）
- 不确定的假设已明确提出而非默默选择（编码前思考）
- 成功标准已定义且已验证（目标驱动执行）
- 调试过程遵循参考实现优先、逐字段对比的纪律（调试纪律）
